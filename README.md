# Модуль "Учет финансов" ТигрБанка.
Этот модуль будет включать инструменты, упрощающие управление личными финансами: анализ доходов и расходов, планирование бюджета, рекомендация выгодных предложений. Такой подход поможет пользователям более осознанно распоряжаться средствами и достигать своих финансовых целей!

## Предметная область 
Программа моделирует часть функциональности личного кабинета клиента банка, отвечающую за хранение, обновление и предоставление информации о счетах, операциях и их категориях; а также предоставляет возможность проводить аналитику по финансовым операциям и сохранять её результаты в файл указанного формата для предоставления по месту требования. Для демонстрации возможностей программного модуля используется реализация конкретного пользовательского сценария в консольном приложении. 

## Архитектура программного модуля 
Система включает в себя:
- набор доменных сущностей, лежащих в основании модуля 
- репозитории - абстракции, имплементирующие CRUD-операции и предназначенные для хранения сущностей 
- набор сервисов/фасадов, которые реализуют некоторую бизнес-логику
- консольное приложение, которое запускает пользовательский сценарий работы.
- юнит-тесты

### Доменный слой 
#### Класс 'Category' 
Моделирует сущность категории операции (например, 'Здоровье', 'Зарплата', 'Кэшбек' и любые другие, которые будет поддерживать банк). Имеет следующие свойства:
- UUID id                   # уникальный идентификационный номер 
- CategoryType type         # тип категории ('INCOME', 'EXPENSE')
- String name               # имя категории

#### Класс 'Operation' 
Моделирует сущность банковской операции. Имеет следующией свойства:  
- UUID id                   # уникальный идентификационный номер 
- OperationType type        # тип операции ('INCOME'/'EXPENSE') 
- UUID accountId            # ссылка на банковский счет
- UUID categoryId           # ссылка на категорию операции
- BigDecimal amount         # сумма операции
- LocalDate date            # дата проведения операции
- String description        # описание операции

#### Класс 'BankAccount' 
Моделирует сущность банковского счета. Имеет следующие свойства: 
- UUID id                   # уникальный идентификационный номер
- String name               # имя счета 
- BigDecimal balance        # сумма счета 


### Слой доступа к данным 
Для каждой сущности из доменного слоя есть интерфейс с CRUD операциями и одна его реализация, предоставляющая возможность хранения сущностей в структуре ConcurrentHashMap<T>, где T - соответствующий тип доменной сущности. 
### Слой бизнес логики 
Сервисы представляют собой абстракции для организации и агрегации специализированных бизнес-процессов. Они инкапсулируют правила предметной области, а следовательно, координируют взаимодействие между репозиториями и доменными сущностями.
#### Класс 'BalanceRecalculator'
Моделирует бизнес-процесс по перерасчету баланса, имплементирует интерфейс 'IBalanceRecalculator'. В качестве зависимости принимает в своей конструктор ссылки на объекты классов, имплементирующих интерфейс репозиториев операций и банковских счетов. В классе реализован метод, который по указанному счёту пересчитывает баланс по всем операциям. При этом логика взаимодействия с репозиториями ложится на его плечи.  
#### Класс 'AnalyticsService' 
Моделирует бизнес-процесс по проведению и предоставлению аналитики, имплементирует интерфейс 'IAnalyticsService'. В качестве зависимости в своей конструктор принимает ссылки на объекты классов, имплеменитрующих интерфейс репозиториев операций и категорий. Аналогично на его плечи ложится логика по взаимодействию с репозиториями для извлечения необходимых данных и проведению аналитических расчетов за выбранный период времени. 
#### Класс 'OperationDataService'
Класс остался от предыдущего ДЗ. Посльку он уже исполнял роль фасада, то код переехал в OperationFacade с некоторыми правками. 
### Экспорт/ импорт 
Возможность сохранения и восстановления состояния репозиториев предоставляют следующие три класса: JsonDataService, YamlDataService и CsvDataService. Все три являются наследниками асбтрактного класса DataService, который в свою очередь реализует паттерн шаблонный метод. Каждый из сервисов переопределяет один из шагов алгоритма импорта, связанного со спецификой типа файла, из которого производится восстановление. 
Стоит отметить, что при экспорте и соответственно последующем импорте, связанными с форматом csv, происходит создание и чтение трех файлов accounts.csv, categories.csv, operations.csv. 
Для того, чтобы соблюсти принцип открытости/закрытости и добавить возможность добавлять какие-либо новые форматы файлов была добавлена абстрактная фабрика и ее реализация: DataServiceFactory и DataServiceFactoryImpl. При этом реализация хранит словарь, ключами которого являются типы файлов, которые поддерживаются данной фабрикой, а значениями являются лямдбды, имплементирующие интерфейс Supplier и позволяющие отложить создание соответствующей реализации DataService.

### Фасады 
Поскольку выполнение различных пользовательских сценариев может требовать скоординированной и последовательной работы различных сервисов, были выделены так называемый фасады, которые играют роль интерефейсов сложной системы.  
**Пример 1**. Рассмотрим конкретный сценарий открытия счета в нашем приложении. Клиенту необходимо передать все сведния необходимые для открытия счета в интерфейс фасада. От него скрыты все дальнейшие манипуляции, и он не знает, что происходит за кулисами. Для клиента важен результат - октрытие счета. В самом же приложении необходимо создать соответствующий объект банковского счета, вызвать метод репозитория для сохрания данных, а также создать банковскую операцию с суммой, которую клиент пожелал положить на счет и сохранить ее в репозиторий (в противном, случае при пересчете баланса по операциям, первичное пополнение не будет учтено, т.к. соответствующей ему операции не существует). 
Аналогично и для закрытия счета. Для того, чтобы закрыть счет, нужно проверить его баланс. Нельзя закрыть счет, если у него ненулевой баланс. Нужно перевести деньги, а только потом закрывать счет, что для нас значит удалить его из репозитория.  
**Пример 2**. При добавлении операции необходимо пересчитывать баланс у счета, к которому привязана эта операция, а также типа этой операции. То есть, такой сценарий требует обращения к несольким сервисам: репозиторию операций, репозиторию счетов и к сервису пересчета баланса. Все эти действия инкапсулирует один из методов, класса OperationFacade. 

### Пользовательские сценарии 
Для проигрывания различных сценаривев приложения реализован паттерн "Команда". Объявляется интерфейс Command с двумя методами execute() и getName(). Классы имплементирующие данный интерфейс, описывают сценарии использования в методе execute(). Метод getName() необходим для идентификации конкретного сценария. 
Класс CommandInvoker объявляет публичный метод runCommand(Command) с аргументом - ссылкой на объект, имплементирующий Command. Выполняет запуск сценария, а также выводит время работы каждой запущенной команды. 
Для демонстрации функциональных возможностей написаны 5 сценариев: 
- InitializeUserFinanceCommand создает два банковских счета и ряд отслеживаемых категорий и выводит состояние своих репозиториев (демонстрация создания и сохранения счетов и категорий)
- CreateAndApplyOperationsCommand создает счета, категории, а также операции. Выводит состояние счетов до операций и после(демонстрация пересчета банковского счета в зависимости от типа операций)
- ShowStatisticsCommand создает счета, категории и операции. После чего выводит статитстика по каждому счету. В том числе и статистику затрат и пополнений по категориям. (демонстрация расчета статистики)
- ExportToFileCommand создаются счета, категории и операции, затем экспортируются в файлы трех типов: json, yaml, csv. (демонстрация экспорта)
- RepairFromFileCommand восстанавливает состояние репозотриев из файлов трех приведенных выше типов. При этом демонстрируется состояние репозиториев до и после каждого восстановления.  (демонстрация импорта)

### Какие из паттернов GoF были реализованы? 
- Абстрактная фабрика. Как и было сказано выше, абстрактная фабрика объявлялась для создания фабрик, которые поддерживают создание объектов для экспорта и импорта, поддерживающих индвидуальный набор типов файлов. 
- Шаблонный метод. Класс DataService объявляет шаблонный метод импорта из файла. Шаблонный метод описывает алгоритм импорта, один из которых является абстрактным, а именно parseFromFile, который переопределяют наследники класса в зависимости от типа файла, с которым работают. 
- Фасад. Были описаны выше. 
- Команда. Были описаны выше. 
Также одним из предложений в задании было использование фабричного метода для изоляции лоигики инициализации объектов. Для изоляции логики инициализации доменных сущностей конструктор этих сущностей был сделан приватным, а также объявлен публичный статический метод create() с необходимыми для инициализации параметрами. В случае успеха он возвращает ссылку на объект сущности. Этот статический метод берет на себя всю логику по инициализации объекта и валидации параметров, требуемых для этой инициализации. Стоит отметить, что такой подход реализует паттерн статическая фабрика, однако он не является полноценной реализацией фабричного метода, и поэтому вынесен за скобки. 

### Почему выбранные решения улучшили качество дизайна?  
#### Паттерн Command
Использование полезно, поскольку объявляет общий интерфейс и единую точку запуска. Сейчас интерфейс Command реализуют только сценарии для демонстрации функциональных возможностей системы, но в дальнейшем это могут быть любые другие более сложные сценарии работы приложения. При этом для создания нового пользовательского сценария не потребуется изменять уже написанную логику - достаточно просто создать новый класс, имлементирующий интерфейс Command. Таким образом, мы соблюдаем DIP и OCP. 
#### Паттерн Facade 
Использование паттерна улучшили архитектуру, поскольку предоставляют достаточно тонкий интерфейс для работы с модулями приложения, переводят их на понятный для пользователя интерфейса язык и инкапсулируют внутреннюю логику взаимодействия модулей системы. Очевидно, из этого следует, что использование данного паттерна уменьшает связность кода. Вместо того, чтобы вручную координировать низкоуровневые сервисы в клиентском коде, увеличивая таким образом связность, нам предоставлен достаточно тонкий интерфейс, в котором изолирована вся логика взаимодействия. Следовательно, мы соблюдаем LowCoupling принцип и HighCohesion принцип(если интерфейс действительно тонкий, т.е. фасад достаточно узкоспециализирован). 
#### Паттерн Template method 
Использование интерфейса для импорта/экспорта в различные типы файлов при следованиии принципам SOLID и GRASP напрашивалось и так: объявить интерфейс или абстрактный класс и реализовать нужные методы для каждого типа файлов. Однако, такой подход породил бы много повторяющегося кода. Шаблонный метод, как раз избавляет нас от его написания. При импортировании есть небольшая часть логики специфичная для конкретного типа файлов, поэтому логично переопределять только ее. 
#### Паттерн FactoryMethod
Как уже было сказано выше, использование абстрактной фабрики для создания объектов DataService по экспорту/импорту в файлы обусловлено следующим соображением. Для того, чтобы добавить поддержку какого-либо нового типа файлов, пришлось бы править клиентский код и это нарушило бы OCP. Использование данного паттерна, позволяет нам создать новую фабрику, которой делегировано создание объектов с уже добавленным новым типом. Именно это и позовляет нам следовать OCP. 

Использование паттернов помогло в том числе следовать принципам SOLID и двум главным принципам GRASP: LowCoupling и HighCohesion.

## Проблемы при расширении 
- Проблема мультивалютности. Сейчас работаем только с одной валютой. Храним баланс как BigDecimal без привязки к валюте. Придется править классы.  
- Отсутствие механизма транзакций. При переводе со счета на счет деньги могут быть сняты, а при сбое не успеют зачислиться. Получаем потерю данных. Необходимо менять бизнес-логику.
